# CashFlow 自动化测试体系说明文档

## 一、概述

CashFlow 的测试体系分为**三层**：

```
┌────────────────────────────────────────────────────────┐
│  Layer 3: 测试报告输出                                  │
│  逐回合数据表 + Bug 日志 + exit code                    │
├────────────────────────────────────────────────────────┤
│  Layer 2: 测试校验器（Validator Pipeline）               │
│  ├─ state_machine → 状态机死锁/卡死检测                 │
│  ├─ bankruptcy   → 破产未触发/变卖未弹窗                │
│  ├─ financial    → 现金异常/数据越界                    │
│  ├─ branch_missing → 分支覆盖缺失（格子/卡牌未进入决策）│
│  ├─ deadlock     → 同一阶段停留过久                     │
│  ├─ card_stuck   → CARD_DECISION 无卡片                 │
│  └─ data_invalid → 子女/幸福度/性别等字段异常            │
├────────────────────────────────────────────────────────┤
│  Layer 1: 自动化 Agent（模拟玩家决策）                   │
│  ├─ 掷骰子 → ROLL_DICE + MOVE_PLAYER                   │
│  ├─ 格子上决策 → 结婚/生育/升迁/慈善/税务              │
│  ├─ 机会卡 → 估值+ROI 判断买入或跳过                   │
│  ├─ 市场卡 → 危机/衰退/折扣/要约收购判定               │
│  ├─ Doodad卡 → 统一 PAY_DOODAD                        │
│  ├─ TURN_END → 还债优先/强制变卖/贷款兜底              │
│  └─ 强制变卖 → 协商变卖或跳过                         │
└────────────────────────────────────────────────────────┘
```

## 二、运行方式

### 2.1 基本命令

```bash
# 运行 50 回合测试（默认）
npx tsx scripts/runAutoTest.ts

# 运行 100 回合
npx tsx scripts/runAutoTest.ts 100

# 运行 200 回合
npx tsx scripts/runAutoTest.ts 200
```

### 2.2 退出码

| 退出码 | 含义 |
|--------|------|
| `0` | 无 critical 级别 Bug |
| `1` | 存在 critical 级别 Bug |

## 三、核心运行流程

### 3.1 入口 → `runAutoTest()`

```typescript
function runAutoTest(maxRounds: number, seed?: number): GameState
```

1. **初始化游戏** — 调用 `gameReducer` 的 `SETUP_GAME` 动作
   - 职业: `engineer`（工程师）
   - 城市: `shanghai`（上海）  
   - testMode: `true`（开启校验器）
   - testMaxRounds: maxRounds

2. **主循环** — `while (phase !== 'GAME_OVER' && !testStopped && steps < maxSteps)`
   - `autoStep(state)` → 根据当前 phase 自动决策
   - 每回合记录 `snapshotHistory`（现金/现金流/支出/工资/被动收入/负债数/房租）
   - 死循环保护：最大步数 = `maxRounds × 50 × 3`
   - 卡死检测：state 未变化 → 打印诊断信息 + break

3. **输出** — 打印逐回合数据表 + Bug 日志

### 3.2 自动决策引擎 → `autoStep()`

根据 `state.phase` 分派决策逻辑：

```
state.phase
├── ROLLING / FAST_TRACK → rollAndMove()
│   └── 掷骰子(ROLL_DICE) → 移动(MOVE_PLAYER)
│
├── CARD_DECISION
│   ├── pendingLiquidation → 强制变卖
│   ├── pendingCashFlowSettlement → 确认现金流
│   ├── pendingLifeEvent === 'retirement' → 确认退休
│   ├── space === 'settlement' + pendingSettlement → 确认税务结算
│   ├── space === 'promotion' + careerEvent → 晋升/跳槽/裁员决策
│   ├── space === 'charity' → 拒绝捐款
│   ├── space === 'marriage' → 结婚/婚姻咨询决策
│   ├── space === 'baby' → 生育/暂缓决策
│   ├── card.type === 'opportunity' → shouldBuyOpportunity()
│   ├── card.type === 'doodad' → PAY_DOODAD
│   └── card.type === 'market'
│       ├── effect.type === 'buyout' → 要约收购（卖出最差资产）
│       ├── effect.type === 'discount' → 抽打折机会卡
│       └── 其他 → APPLY_MARKET_EFFECT
│
└── TURN_END
    ├── cash < 0 + 有可变卖资产 → 强制变卖
    ├── cash < 0 + 无可变卖资产 → 贷款兜底
    ├── 持有严重高估股票 → 卖一半（概率40%）
    └── 有高优先级负债 + 现金充足 → 还款
        └── 最后 → END_TURN
```

### 3.3 购买机会卡策略 → `shouldBuyOpportunity()`

```typescript
function shouldBuyOpportunity(player, card, marketMultiplier, sectorMultiplier, recentCrisis, recentGrowth): boolean
```

**决策树**（按优先级）：

```
1. 近退休（age ≥ 45 或退休 ≤10年）
   ├── 正现金流被动资产(realEstate/reit/bond/stock) → ✅ 买
   └── 商铺 ROI < 3% → ❌ 不买

2. 股票且现金够买1手 → ROI > 0.3% 才买

3. 近期有危机
   ├── 债券/大宗商品 → ✅ 买
   ├── 防御型行业(金融/公用/消费/利率债/贵金属) → ROI > 1% 才买
   └── 成长型行业(科技/新能源/先进制造) → ❌ 不买

4. 近期有增长 + 成长型行业 → ✅ 买

5. 防御型行业 → ROI > 2% 才买

6. 债券/REITs → ROI > 2% 才买

7. 默认 → ROI > 5% 才买
```

**ROI 计算**：

```
股票 ROI = (年股息 × 100) / (单价 × 12)     → 月股息率
实体 ROI = 月现金流 / 首付                    → 月回报率
```

## 四、Bug 检测体系（Validator Pipeline）

### 4.1 怎么触发的

在 `GameReducer.ts` 的 `gameReducer` 函数末尾：

```typescript
export function gameReducer(state: GameState, action: GameAction): GameState {
  const nextState = gameReducerSwitch(state, action);
  return runTestValidators(nextState, state, action);  // ← 每次 dispatch 后自动校验
}
```

每次 `dispatch(action)` → reducer 执行逻辑 → **自动调用 `runTestValidators`**。

### 4.2 检测项完整列表

| 校验器 | 触发条件 | 严重级别 | 描述 |
|--------|---------|---------|------|
| **state_machine** | MOVE_PLAYER 后 pendingDice 未清除 | critical | 状态机残留骰子数据 |
| **state_machine** | 超过 testMaxRounds 未 GameOver | warning | 游戏超时未结束 |
| **deadlock** | 同一 phase 连续停留 >5 次 | critical | 某阶段死循环卡死 |
| **card_stuck** | CARD_DECISION 但 currentCard=null 连续 >3 次 | critical | 无卡可显示 |
| **bankruptcy** | 破产条件达成但未进入 GAME_OVER | critical | 破产判定失败 |
| **bankruptcy** | 现金耗尽但可变卖资产未弹窗 | critical | 变卖窗口缺失 |
| **financial** | cash < -1,000,000 | critical | 现金极度负值 |
| **financial** | cash > 1,000,000,000 | warning | 现金异常高 |
| **financial** | 单次现金变动 >50,000,000 | warning | 大额异常交易 |
| **financial** | children < 0 或 >10 | critical | 子女数量越界 |
| **financial** | marriageHappiness < 0 或 >100 | critical | 幸福度越界 |
| **branch_missing** | 已婚玩家在 baby 格但未进 CARD_DECISION | critical | 生育分支遗漏 |
| **branch_missing** | 婚恋格未进 CARD_DECISION | critical | 婚姻分支遗漏 |
| **branch_missing** | 税务结算格未进 CARD_DECISION | critical | 税务分支遗漏 |
| **branch_missing** | REPAY_LIABILITY 后 phase 异常 | warning | 还款后状态异常 |
| **branch_missing** | 子女费用卡在无子女玩家触发 | critical | 卡牌身份校验缺失 |
| **branch_missing** | 配偶失业卡在未婚玩家触发 | critical | 卡牌婚姻校验缺失 |
| **branch_missing** | 晋升事件在退休/失业玩家触发 | critical | 事件身份校验缺失 |
| **branch_missing** | 高负债未扣幸福度（结婚玩家） | warning | 负债惩罚遗漏 |
| **data_invalid** | 非女性有产假 | critical | 产假性别校验 |
| **data_invalid** | 女性0-3子嗣幸福度超高 | warning | 育儿幸福度异常 |
| **data_invalid** | childAges 超出 0-36 范围 | critical | 子嗣年龄越界 |
| **data_invalid** | 已婚无 familyIncome | warning | 家庭收入缺失 |

### 4.3 Bug 数据结构

```typescript
interface BugLogEntry {
  id: string;
  timestamp: number;
  round: number;
  playerId: string;
  category: 'state_machine' | 'deadlock' | 'card_stuck' | 'bankruptcy' | 'financial' | 'branch_missing' | 'data_invalid';
  severity: 'critical' | 'warning';
  message: string;
  action?: string;           // 触发该 Bug 的 action.type
  snapshot?: Record<string, unknown>;  // 现场数据快照
}
```

## 五、Bug 排查方法

### 5.1 快速诊断流程

```
Bug 出现
│
├─ category = financial
│   └─ 查看逐回合数据表：现金从哪回合开始异常
│       ├─ 骤降 → 找对应回合的 action
│       └─ 骤升 → 检查买入/卖出计算
│
├─ category = bankruptcy
│   └─ 查看 cash + cashFlow 趋势
│       ├─ cash=0 + cashFlow 为负 + 有资产 → 变卖弹窗未出
│       └─ cash=0 + cashFlow 为负 + 无资产 → GameOver 应触发
│
├─ category = deadlock / card_stuck
│   └─ 看 snapshot.phase + currentCard
│       ├─ phase = CARD_DECISION + currentCard = null
│       │   → autoStep 缺少对应空间类型的处理分支
│       └─ phase = TURN_END
│           → autoStep 中 TURN_END 分支缺少 fallback
│
├─ category = branch_missing
│   └─ snapshot.phase 显示游戏阶段
│       ├─ 格子类型 → reducer 中对应 space.type 未正确处理
│       └─ 卡牌类型 → 卡牌数据中 filterConfig 不完整
│
└─ category = data_invalid
    └─ message 指出具体字段
        → reducer 中对应字段的初始化或更新逻辑有误
```

### 5.2 逐回合数据表解读

测试输出的数据表每列含义：

```
回合    现金        现金流      支出       工资       被动收入    负债数  房租    阶段
 1     10,679      2,679      11,240     14,490      -571      3       0     CARD_DECISION
```

| 列 | 含义 | 异常信号 |
|----|------|---------|
| 回合 | 游戏进行轮次（非步数） | — |
| 现金 | 当前持有现金 | 连续 0 或骤变 |
| 现金流 | 每月净现金流 | 从正突然转负且不再恢复 |
| 支出 | 月总支出 | 突增 |
| 工资 | 月工资收入 | 减半 → 失业 |
| 被动收入 | 资产带来的月收入 | 骤降 → 市场事件影响 |
| 负债数 | 贷款笔数 | 突然增多 → 兜底贷款爆发 |
| 房租 | 月租金 | >0 且自住房存在 → Bug |

**典型异常模式**：

```
模式1: 现金持续下降 → 现金流为负 → 负债数不断增多
      → 债务螺旋，检查 getFixedExpenses 是否重复计算

模式2: 现金第 1 回合就为负
      → 初始数据异常，检查 createPlayer 的 cash 计算

模式3: 现金流突然从+变-且房租从0变>0
      → 自住房标记丢失，检查 isSelfLiving 逻辑

模式4: 现金巨额增长(>1000万)
      → 卖出资产未扣除贷款，检查 settleAssetLoan
```

### 5.3 常见 Bug 根因定位

| Bug 现象 | 检查文件 | 检查函数/行 |
|----------|---------|------------|
| 开局暴毙 | `GameReducer.ts` | `createPlayer` — 初始资产/负债/房租 |
| 现金负数 | `financial.ts` | `ensureCashNotNegative`, `wrapCashGuard` |
| 现金流计算异常 | `financial.ts` | `getMonthlyCashFlow`, `getFixedExpenses` |
| 卖出资产现金异常 | `GameReducer.ts` | `SELL_ASSET`, `settleAssetLoan` |
| 卡牌身份过滤失效 | `doodadCards.ts` | `filterConfig` 字段 |
| 状态机卡死 | `GameReducer.ts` | `CONFIRM_CASH_FLOW_SETTLEMENT` |
| 房租错误 | `financial.ts` | `getRentExpense` |
| 房产/车辆不显示 | `AssetCenter.tsx` | 资产分组渲染逻辑 |

## 八、新增测试规则指南

如需新增一条 Bug 检测规则，只需三步：

**Step 1**: 在 `testValidators.ts` 的 `runTestValidators()` 中添加校验逻辑：

```typescript
if (some_condition) {
  next = appendBug(next, {
    category: 'your_category',  // 已有: state_machine | deadlock | card_stuck | bankruptcy | financial | branch_missing | data_invalid
    severity: 'critical',       // critical 导致 exit code 1
    message: '描述 Bug 现象的具体文字，带上关键数值',
    action: action.type,
    snapshot: { key: value },   // 可选：现场数据快照
  });
}
```

**Step 2**（可选）：在 `GameState` 中添加新字段用于追踪状态

**Step 3**：运行测试验证新规则可被触发

## 九、常见问题 FAQ

**Q: 测试显示 STUCK no state change 怎么办？**
A: 说明 `autoStep()` 的某个分支返回了 `state` 本身（即 dispatch 没有改变任何状态）。查看输出的 phase/space/card 信息，在 `autoStep()` 中找到对应 phase 分支补全决策逻辑。

**Q: Bug 日志显示 "高负债幸福惩罚缺失" 但这不是现在要改的功能？**
A: 这是 `warning` 级别的已知功能缺口，不影响 `exit code`。属于待办事项。

**Q: 测试显示 Game Over 但数据正常？**
A: Game Over 本身不一定是 Bug。查看逐回合数据表的被动收入和负债数变化趋势，如果是被动收入不足以覆盖支出导致的自然破产，属于正常的游戏平衡表现。如果 5 回合内就破产且未购买任何资产，检查初始资产/负债配置。
